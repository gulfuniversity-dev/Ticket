# 🎓 Gulf University — Graduation Ceremony E-Ticket & QR Check-in System

A complete system that imports existing student ticket-purchase data, generates a
**unique secure QR e-ticket** for every purchased ticket, produces printable PDF
tickets, and controls guest entry at the ceremony gate. Each QR is valid for
**one entry only** — re-scanning shows **“Already Used”** and rejects entry.

> The system does **not** handle payment. It works from the ticket data you already have.

---

## ✨ Features

- **Admin login** (secure, role-based) and **Gate Staff / Scanner** login.
- **Import** an Excel/CSV file of students and ticket counts → auto-generates the
  correct number of tickets, each with a unique cryptographically-secure QR token.
- **PDF e-tickets** — per-student or all students, with QR codes and the
  "valid for one entry only" instruction.
- **Gate scanner page** — works on mobile / tablet / laptop cameras, with large
  green / red / yellow result messages and a scan **cooldown** to stop duplicate detections.
- **Atomic one-time check-in** — impossible to use the same ticket twice, even under a race.
- **Manual search** by Ticket ID / Student ID / Student Name (camera fallback).
- **Admin dashboard** — totals, live attendance, recent check-ins, duplicate-scan attempts.
- **Ticket management** — cancel, reactivate, manually mark used, undo last check-in.
- **Reports** — export check-in log and tickets to Excel/CSV; full audit log of every action.

## 📥 Importing Data

### A) The two source files (this dataset)

`npm run db:import` reads the two spreadsheets in `data/`:

- **`GraduationFeesReport.xlsx`** — every graduating student. Each gets **1 base ticket**.
- **`AdditionalTickets.xlsx`** — students who bought **extra** tickets.

**Total tickets per student = 1 (base) + additional purchased.**
For the provided data this is **174 students → 632 tickets**.

To point at different files:

```bash
node scripts/import-data.mjs <fees.xlsx> <additional.xlsx>
```

### B) Admin upload (single normalized file)

The **Admin → Import** page accepts one Excel/CSV file with these columns
(see `data/sample-import.xlsx`, generated by `npm run db:sample`):

| StudentID | StudentName | Email | Phone | TicketsPurchased |
|-----------|-------------|-------|-------|------------------|
| 200104115176 | AYSHA MOHAMED … | aysha@example.com | +97333000001 | 5 |

- `Email` and `Phone` are optional.
- If a student bought 4 tickets, **4 tickets with 4 unique QR codes** are created.
- Re-importing the same student **tops up** to the new count without duplicating tickets.

## 🖥️ How It Works (Workflow)

1. **Admin** imports the student/ticket data → system generates unique QR tickets.
2. **Admin** downloads (or emails*) the PDF e-tickets to students.
3. On graduation day, **Gate Staff** open `/scan` and scan each guest's QR.
4. A valid unused ticket → **green “Valid Ticket – Entry Allowed”**, marked used, time + staff logged.
5. A second scan of the same ticket → **red “Already Used – Entry Not Allowed”** (shows previous time).
6. Unknown QR → **yellow “Invalid Ticket”**. Cancelled ticket → **red “Cancelled Ticket”**.

\* Email delivery is an optional extension point — see *Optional: Email* below.

## 🔐 Security

- **QR tokens are 256-bit cryptographically-random** (`crypto.randomBytes(32)`), not derived
  from the student ID — they cannot be guessed or forged.
- **One-time use** is enforced by a single atomic SQL statement
  (`UPDATE … WHERE qr_token=$1 AND status='unused' RETURNING …`), so concurrent
  double-scans are impossible.
- **Role separation**: gate staff can only scan and search — they cannot edit ticket data.
  Only admins can cancel, reactivate, mark-used or undo. Enforced in both middleware
  and every API route.
- Passwords are stored as **bcrypt** hashes. Sessions are signed JWTs in **httpOnly,
  SameSite=Lax** cookies (12h expiry).
- **Every gate action is logged** to `check_in_logs` (valid_entry, already_used,
  invalid_ticket, cancelled_ticket, manual_checkin, undo_checkin, cancel, reactivate)
  with staff id and device user-agent.
- The scanner applies a **3-second cooldown** + same-token de-dupe to prevent repeated
  camera detections from logging duplicates.

> Camera access requires **HTTPS** (or `localhost`). On a deployed URL this is automatic.

## 🗄️ Database Schema

See [`db/schema.sql`](db/schema.sql). Four tables:

- **users** — `id, name, email, password (bcrypt), role (admin|gate_staff), …`
- **students** — `id, student_id (unique), student_name, email, phone, tickets_purchased, …`
- **tickets** — `id, ticket_id (e.g. GU2026-000123), student_id, qr_token (unique secure),
  ticket_number, total_tickets, status (unused|used|cancelled), used_at, checked_in_by, …`
- **check_in_logs** — `id, ticket_id, student_id, action, scanned_by, scanned_at, device_info`

## ☁️ Deployment

The app is a standard Next.js app — deploy to **Vercel** (recommended) or any Node host.

### Vercel
1. Push this folder to a Git repository.
2. Import it in Vercel.
3. Set Environment Variables (from `.env.local`): `DATABASE_URL`, `JWT_SECRET`,
   `NEXT_PUBLIC_CEREMONY_TITLE`, and optionally the seed-account vars.
4. Deploy. Then run the DB setup once (locally or via `vercel` shell):
   `npm run db:setup`.
5. Open the deployed HTTPS URL — camera scanning works automatically over HTTPS.

### Self-hosted (Node)
```bash
npm ci
npm run build
npm run start            # serves on :3000 (put Nginx/HTTPS in front)
```

**Supabase note:** use the **Session pooler** connection string (port `5432`,
host `aws-<n>-<region>.pooler.supabase.com`, user `postgres.<project-ref>`).
SSL is enabled in code.

## ✉️ Email tickets to students

Tickets are emailed as a PDF attachment to **`<StudentID>@gulfuniversity.edu.bh`**
via **Office 365 SMTP** (`nodemailer`). Every send is recorded in the `email_log`
table and stamps `students.last_emailed_at`.

**Setup** — fill these in `.env.local` (see `.env.example`):
`SMTP_USER`, `SMTP_PASS` (an Office 365 mailbox + app password), `MAIL_FROM`,
and `EMAIL_BULK_SECRET`. The mailbox must have **SMTP AUTH enabled** in the
Microsoft 365 admin center, and `STUDENT_EMAIL_DOMAIN` controls the recipient domain.

**Send to one student:** the **Email tickets** button on a student's admin page
(`POST /api/admin/students/[studentId]/email`).

**Send to everyone (bulk):** start the app (`npm run start`), then run the
throttled script — it drives the same endpoint so it never hits the serverless
time limit:
```bash
npm run send-tickets -- --limit 1   # smoke test: first student only
npm run send-tickets                # everyone not yet emailed
npm run send-tickets -- --force     # resend to everyone
npm run send-tickets -- --dry-run   # list recipients, send nothing
```
